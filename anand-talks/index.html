I. The Repository That Slept

On February 17, 2011, S. Anand made three commits to a GitHub repository called "talks." He added some Python imaging code. Then he walked away.

The repository sat there, dormant, for 5,112 days. Occasionally someone would star it. The internet moved on. GitHub introduced Actions, Copilot, and Codespaces. OpenAI released GPT-2, then GPT-3, then GPT-4. The world changed.

And then, on May 20, 2025, at 4:42 PM Singapore time, Anand made a commit with a simple message:

"Revamp and add new vibe coding talk transcript"

What happened next is the story.

III. The Concept That Changed Everything

On May 10, 2025, Anand gave a talk at SETU school. He introduced a term that would appear in 7 more talks over the next 7 months:

"Vibe Coding"

Here's how he defined it:

"Vibe coding is like AI coding, but with one difference, where you don't check the output. AI gives you code, you run it."

Wait. You don't check the code?

"It sounds a little dangerous because you have no idea what it is going to do, including potentially delete files on your system if you just happily run it. But that is what vibe coding is about."

For anyone trained in software engineering, this is heretical. The entire discipline is built on verification, testing, code review, understanding what the code does.

But then he demonstrates it. Live. On camera. With 65 minutes of unedited video showing:

  1. Asks Claude: "Create a simple web page that takes a picture from my camera"
  2. Gets HTML/JavaScript code
  3. Runs it without reading a single line
  4. Camera permission error
  5. Tells Claude: "Fix the camera error"
  6. Runs new code without reading it
  7. Works

Total time from concept to working application: 22 minutes.

The Core Claim

"Non-programmers can build applications. Programmers can build applications with their mind entirely focused on what needs to be done without worrying about the structure of the code."

IV. The Mathematics of Disposability

Here's why vibe coding works: The cost structure of code has fundamentally changed.

The Old Economics (2000-2023)

  • Writing code: 2-10 hours (expensive)
  • Debugging code: 1-5 hours (expensive)
  • Regenerating code: N/A (impossible)

Rational strategy: Debug the code you have. Sunk costs demand it.

The New Economics (2024-2025)

  • Writing code: 0.5-2 minutes (LLM does it)
  • Debugging code: 5-30 minutes (you do it)
  • Regenerating code: 0.5-2 minutes (LLM does it)

Rational strategy: Don't debug. Regenerate.

"Try once, maybe twice. If it doesn't work well, start again."
— Applied Vibe Coding, November 15, 2025
"Throw it away and redo. It's easier than fixing."
— Data Design by Dialogue, June 27, 2025

This is zero marginal cost thinking applied to code. When something costs nothing to reproduce, you stop trying to repair it. You just make a new one.

Traditional Coding vs. Vibe Coding: Productivity Comparison

The Productivity Implication

Traditional approach: 1 app in 10 hours (1 attempt, refined)

Vibe coding approach: 10 apps in 10 hours (10 attempts, best one wins)

You're not working 10x faster. You're trying 10x more things.

2. LLM Psychology

"I look at how LLMs respond to inputs. And they have very varied personalities."
— November 6, 2025

Personality Testing:

  • GPT-4o: High openness to experience, high agreeableness (Einstein, Tagore, M.F. Husain - curious, innovative)
  • Gemini Flash: Very introverted (A.R. Rahman, J.K. Rowling, Rahul Dravid - reserved, reflective)

The Double-Checking Discovery

  • Single model: 14% error rate
  • Double checking: 3.7% error
  • Triple checking: 2.2% error
  • Quintuple checking: 0.7% error (99.3% accuracy)
  • Manual review reduced to: 28% of original work

That's a 95% reduction in errors just from redundancy.

VI. The Five "Wait, Really?" Moments

LLM Cost Evolution (2023-2025)

  • GPT-4o mini (July 2024): $0.15 per million tokens
  • GPT-5 Nano (2025): $0.05 per million tokens

Implication: What was economically prohibitive 18 months ago is now trivial.

2. The Job Market Paradox

Wait. If AI writes code, shouldn't developer jobs be declining?

"When the cost of something goes down, the demand goes up. A programmer who could write 100 lines of code can now write 300 lines of better quality."

3. The Clinical Trial Validation

Problem: Informed consent forms must comply with ICH E6 guidelines. Manual review takes 2-4 hours per form.

LLM Solution:

  • Upload 39-page ICF document
  • Upload 89-page ICH E6 guideline
  • Prompt: "Check this ICF against ICH E6. Find violations."
  • Result: Found 15 errors in 2 minutes 41 seconds
  • Cost: ~$50 (vs. $500K-$2M annually for manual review)

Every programmer is taught: "Document your code." It's professionalism. It's good practice.

But if AI can generate documentation from code better than you can, why are you wasting time writing it?

VII. What This Means for You

For Individual Practitioners

  • Stop optimizing for mastery of tools — Skills with a 6-month half-life aren't worth deep investment
  • Start optimizing for learning velocity — The skill isn't "knowing Pandas." It's "learning the new tool faster than others"
  • Develop taste — Automated: code, EDA, documentation. Human: Is this insight true? Is this visualization misleading?
  • Document failures publicly — Old model: Present polished work. New model: Show the messy process
  • Keep an impossibility list — Review monthly. What can you do now that you couldn't do last quarter?

For Organizations

  • Hire for adaptability, not credentials — Someone experimenting daily with LLMs in 2025 may be more valuable than someone with a CS degree from 2019
  • Shorten training cycles — If skills decay in 90 days, annual training is useless
  • Value speed over perfection — "Good enough now" beats "perfect in 6 months"
  • Build redundancy into AI systems — The "cross-check five LLMs" pattern should be standard architecture

For Educators

The Hard Question: If what you teach is obsolete in 90 days, what do you teach?

Anand's Answer:

  • Don't teach "how to use Claude." Teach "how to figure out which AI to use for which task."
  • Don't teach "how to write Python." Teach "how to verify AI-generated Python is correct."
"I'm able to detect plagiarism much more easily... And I'm encouraging plagiarism."

The task didn't change. The method changed. And the method changed because the tools changed.

IX. The Question That Remains

If expertise decays in 90 days, and the best strategy is continuous re-learning, what happens to wisdom?

Wisdom is supposed to come from long experience. Patterns you see only after years in a domain. Judgment that comes from making mistakes and learning.

But if the domain changes every 90 days, long experience becomes long obsolescence.

Maybe wisdom in the AI age isn't accumulated knowledge. Maybe it's meta-knowledge: knowing how to learn quickly, knowing when to trust AI vs. verify manually, knowing which tasks to delegate vs. which require human judgment.

If that's true, then this repository isn't a historical record.

It's a training manual for the future.
